One-way sync of a [memos](https://usememos.com) instance to a directory of
markdown files, over the HTTP API. New memos become files, deleted or archived
memos have their files pruned, and unchanged memos are left untouched (their
modification times are preserved). Daily cadence is plenty — it is not real-time.

A multi-arch Docker image is published to the GitHub Container Registry (GHCR).

## What it produces

```
<export_dir>/<username>/<date>_<uid>_<title>.md
```

- `<date>` is `YYYY-MM-DD` from the memo's creation time (default) or update time.
- `<uid>` is the first 8 characters of the memo's public id — enough to keep
  same-date memos from colliding. The full uid is preserved in the frontmatter.
- `<title>` is a human-readable slug derived from the memo's first non-empty
  line (links reduced to their visible text). It is lowercased, hyphenated, and
  truncated to 50 chars. Empty memos are just `<date>_<uid>.md`.

Each file has YAML frontmatter followed by the raw markdown content:

```markdown
---
uid: aBc123
created: 2026-07-10T09:00:00Z
updated: 2026-07-11T09:00:00Z
visibility: PUBLIC
pinned: false
tags:
  - alpha
  - zeta
attachments:
  - filename: "photo.png"
    uid: xY9
    type: image/png
    size: 20480
    url: "/file/attachments/xY9/photo.png"
---
The memo body, verbatim.
```

## Why one access token per user

memos' `ListMemos` **never returns another user's PRIVATE memos, even to the
instance host** — the server hardcodes a filter equivalent to
`creator_id == currentUser.id || visibility in ["PUBLIC","PROTECTED"]`, with no
admin bypass. So a complete mirror that includes private memos requires **one
personal access token per user**.

Each token is used to fetch *only its owner's* memos (scoped server-side by
`creator`), which also guarantees every memo is captured exactly once — instead
of shared public memos showing up under every token.

**Pruning is per-user and only happens on success.** If a token fails (revoked,
network error), that user's existing files are left intact — a failed sync is
never treated as "all that user's memos were deleted." A retired token therefore
leaves a **stale folder** behind (acceptable; just delete it by hand when you
retire a user).

## Minting a personal access token

Each user, logged into memos, goes to **Settings → Access Tokens → Create**, and
copies the token. Collect one token per user you want mirrored and hand them to
the exporter via `MEMOS_TOKENS` or `MEMOS_TOKENS_FILE`.

## Configuration

All configuration is via environment variables.

| Variable | Required | Default | Meaning |
|---|---|---|---|
| `MEMOS_BASE_URL` | yes | – | e.g. `http://memos:5230` |
| `MEMOS_EXPORT_DIR` | yes | – | output root |
| `MEMOS_TOKENS` | one of these | – | newline/comma-separated tokens |
| `MEMOS_TOKENS_FILE` | one of these | – | file of tokens, one per line, `#` comments |
| `MEMOS_DATE_BASIS` | no | `created` | `created` or `updated` |
| `MEMOS_TIMEZONE` | no | UTC | IANA tz for the `<date>` bucket |
| `MEMOS_INCLUDE_ARCHIVED` | no | off | `1` to also mirror archived memos under `_archived/` |
| `MEMOS_VERIFY_TLS` | no | on | `0` to disable TLS verification (self-signed) |
| `MEMOS_INTERVAL_SECONDS` | no | – | if set, loop forever with this sleep; else run once |

**`created` vs `updated`.** With `created` (default), a memo's path stays stable
across edits — good for quiet backups where you don't want files moving around.
With `updated`, editing a memo relocates its file to the new date bucket (pruning
handles the move, so no duplicate is left behind).

**Timezone.** The default is UTC for reproducibility. Set `MEMOS_TIMEZONE` to an
IANA name (e.g. `America/New_York`) to bucket dates in local time — note this can
shift a memo created near midnight into the adjacent day's folder.

**Exit code.** The process exits non-zero if *any* token failed, after syncing
the rest.

## Attachments and finding the file on disk

The memos API returns each attachment inline with the memo (no per-memo call),
exposing `filename`, `uid`, `type` (MIME), `size`, and — for external/S3
attachments — an `externalLink`. It **does not expose the internal on-disk
path**. For LOCAL storage, memos writes the file under
`<data_dir>/<Reference>`, where `Reference` follows an admin-configurable path
template (default `assets/{timestamp}_{uuid}_{filename}`) whose timestamp and
random uuid are chosen at upload time and stored only in the database — the API
never returns them.

So the exporter emits what the API *does* give, per attachment: `filename`,
`uid`, `type`, `size`, and either

- `url` — the server-relative serving path `/file/attachments/<uid>/<filename>`
  (works against a running memos instance), for local attachments; or
- `external_link` — for external/S3 attachments (no `url` in that case).

**Locating the actual file in your assets bind mount:**

- With the **default** template the on-disk filename ends with the original
  `filename`, so glob for it:
  ```
  ls assets/*your-filename.png
  ```
- For clean 1:1 resolution, set the memos storage path template to a
  filename-forward form such as `assets/{filename}` or
  `assets/{year}/{month}/{filename}` (memos Settings → Storage). On filename
  collisions memos appends the attachment `uid` — which is in the frontmatter —
  so the pair (template + `uid`) reconstructs the path. Note only **new** uploads
  adopt a changed template.
- If exact paths are a hard requirement, the only authoritative source is the
  database `attachment.reference` column. That is **out of scope** for this
  API-only tool and mentioned here only as an escape hatch.

## Running with Docker

Pull the published image (replace the tag as needed):

```bash
docker pull ghcr.io/clnhlzmn/memos-markdown-exporter:latest

docker run --rm \
  -e MEMOS_BASE_URL=http://memos:5230 \
  -e MEMOS_EXPORT_DIR=/export \
  -e MEMOS_TOKENS="tok_alice,tok_bob" \
  -v /path/on/host/export:/export \
  ghcr.io/clnhlzmn/memos-markdown-exporter:latest
```

> The GHCR package is **private until you flip it public** in the repository's
> package settings after the first publish. Until then, `docker login ghcr.io`
> with a token that has `read:packages` before pulling.

**Volume permissions.** The container runs as root by default so it can write to
a volume owned by any host uid. To run as a specific user (Unraid PUID/PGID
style), pass `--user 1000:1000` — the export directory must then be writable by
that uid/gid.

### docker compose (on-demand + scheduled)

Run it as an on-demand tool on the same network as memos:

```yaml
services:
  memos:
    image: neosmemo/memos:stable
    volumes:
      - ./memos-data:/var/opt/memos
    # ...

  memos-export:
    image: ghcr.io/clnhlzmn/memos-markdown-exporter:latest
    profiles: ["tools"]          # not started by `up`; run on demand
    environment:
      MEMOS_BASE_URL: http://memos:5230
      MEMOS_EXPORT_DIR: /export
      MEMOS_TOKENS_FILE: /run/secrets/memos_tokens
      MEMOS_TIMEZONE: America/New_York
    volumes:
      - ./export:/export
      - ./tokens.txt:/run/secrets/memos_tokens:ro
```

Run a one-off sync:

```bash
docker compose run --rm memos-export
```

**Daily scheduling.** Either:

- **Host cron / [ofelia](https://github.com/mcuadros/ofelia)** invoking
  `docker compose run --rm memos-export` once a day (recommended — the container
  exits between runs); or
- Set `MEMOS_INTERVAL_SECONDS=86400` and drop `profiles: ["tools"]` so the
  service stays up and loops internally.

## Behavior notes

This tool is a faithful repackaging of a tested reference implementation; its
sync behavior is unchanged. It is **one-way** — nothing is written back to
memos. Attachment blobs are **not** copied (they already live in the memos assets
directory; we only reference them by name/path). No database access — API only.
